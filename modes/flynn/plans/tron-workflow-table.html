Standing layer

PULSE — the dispatch loop

Re-runs at tron:start and on every pulse return.

  1. Keep every worker slot busy — follow this workflow in tandem with SWITCHBOARD.
  2. Anything unexpected → consult / follow SCRIPTS (most-specific match wins; * is the fallback).
  3. Proceed to SWITCHBOARD.

SWITCHBOARD — selection

Statuses: pending · cleared · in-progress · blocked · done · abandoned. Only cleared is dispatchable. Run all four, in order:

  1. Fill slots — while a slot is free & cleared work exists, dispatch one per slot, then flip cleared→in-progress:
    • oldest open cleared adhoc (by ID)
    • a due cadence reviewer (consume its counter)
    • next cleared block by sequential ID
  2. Clear ahead — enqueue the architect (review:next:<block>) for every pending block not already queued.
  3. Wait — no free slot, or all remaining in-progress/queued/blocked → re-enter on next pulse.
  4. Session end — nothing pending/cleared/in-progress/queued, no adhoc, no cadence, all idle → session:end. Blocked blocks hold the run open.

Trigger grammar — lowercase, colon-delimited, no spaces. Forms: domain:object (dispatch) · subject:event · subject:object:event.

subjects: tron · build · block · review · cadence · session · worker · operator · wall  ·  events: build · done · clear · stalled · next · end · adhoc · raised · decision

<type> ∈ {code, security, data, …} · <block> = pipeline ID (NN-NN-NN) · next = the next dispatch target (reserved) · * = wildcard · | = alternatives · terminals: end, —.  pulse is the dispatcher control signal, not a table row.
